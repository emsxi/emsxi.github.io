CDN加速

CDN加速全称Content Delivery Network，内容分发网络，是对域名访问的加速
目的：
为了解决因地域分布、带宽、服务器性能带来的访问延迟和低可用性问题，使用户可就近取得所需内容，提高用户访问网站的响应速度和成功率
CDN在全球（或者某个大范围内）分布加速节点，实时地根据网络流量和各节点的连接、负载状况以及到用户的距离和响应时间等综合信息将用户的请求重新导向离用户最近的服务节点上，避免多次网络转发

主要使用场景：
静态资源加速（图片,js,css）、流媒体（直播，点播，监控）

实现方式：
核心是在浏览器和服务器之前增加cache层

  • 用户在浏览器中输入域名
  • 浏览器向DNS发出解析请求，获得对应的ip地址
    • 在本地hosts文件中查找
    • 本地DNS缓存
    • 本地DNS服务器请求
    • 配置转发器，向上递归查询，直到GSLB，然后返回最佳CDN接入点的ip(根据ip的地理位置，网络类型等)，并放入本地DNS的缓存
  • 浏览器请求该CDN节点
  • CDN节点经过内部路由，最后一个CDN节点转发请求至目标服务器
  • 服务器响应，并回传数据
  • 各个CDN节点获取数据并缓存，下次请求进来就无需到目标服务器，可以直接返回结果，对静态资源加速尤为显著
  • 最后一个CDN节点回传数据给用户

构成：
中心节点：一台负责GSLB的DNS服务器，以及网管中心
网管中心：监控各个设备的流量，告警等等，保证高可用
GSLB：通过某种策略，返回最近的CDN节点ip，并保证节点可用且健康
边缘节点：各个节点的cache，每个节点可以有多台cahce，通过负载均衡器协同工作
cahce:高速缓存服务器

传统网站访问流程

  • 用户在浏览器中输入域名
  • 浏览器向DNS发出解析请求，获得对应的ip地址
    • 在本地hosts文件中查找
    • 本地DNS缓存
    • 本地DNS服务器请求
      • 本地区域配置资源返回，权威的
      • 缓存的映射关系，非权威的
    • 向上层寻找
      • 如果配置了转发器，就一级级向上寻找
      • 如果没有配置转发器，发送13台root服务器，根据根域名获得对应的ip，请求该ip，有结果就return，否则return下一级dns直到找到
  • 浏览器请求该ip地址
  • 服务器响应，并回传数据

负载均衡

目的：
将工作负载分布到多个服务器来提高网站、应用、数据库或其他服务的性能（例如吞吐量）和可靠性，包括流量，数据等等
避免单点过载、崩溃等单点故障

主要使用场景：
自动分配流量

  • ALB：运行于application层，主要为http,https设置转发规则
  • NLB: 运行于network层，主要为tcp和udp设置转发规则
  • CLB: ALB+NLB

实现方式：

  • 管理员配置转发规则，即负载均衡算法
  • 先判断是否可用，健康检查服务定期尝试使用转发规则定义的协议和端口去连接后端服务器
    • 服务器无法通过健康检查，就会从健康服务器池（healthy pool）中剔除，保证流量不会被转发到该服务器，直到其再次通过健康检查为止。
    • 通过就是可用的
  • 然后用规则选择服务器
    • 轮询，适用于平均分布的情况
    • 加权轮询，考虑节点差异性的轮询，例如计算密集型考虑CPU和内存，IO密集型考虑磁盘性能
    • 哈希，用ip生成散列表，适用于用户和服务器要保持seesion，或者是读取某个存储的情况（有状态的请求），但是可能会出现粘性连接的问题
    • 最小连接，优选连接数少的服务器，适用于会话普遍较长的情况
  • 解决单点故障
    • 多个负载均衡器连接成一个集群
    • 主故障，转发给备，同时使用浮动ip的策略，使得ip可以在服务器间移动，解决ip地址重新映射的问题（浮动ip还适用于主从切换的场景）

构成：
负载均衡器可以
- 放在客户端自己选择
- 服务器列表前防止集中式代理proxy负责分发，例如nginx

负载均衡与消息队列的区别：
一个是主动推送，实时，但是如果大量请求可能会崩溃
一个是被动拉取，不实时，但是稳定，而且新增删除节点更加容易

各种网络协议

https如何保证安全

  • 对称协议加密数据，但是密钥会泄漏
    • 服务器公钥加密数据
    • 客户端公钥解密数据
  • 使用非对称协议加密数据，但是可以被篡改或者影响性能
    • 服务器发送公钥和公钥加密数据
    • 客户端生成私钥，用私钥解密
  • 合并对称和非对称
    • 服务器发送公钥
    • 客户端生成一把对称加密用的密钥
    • 客户端用公钥加密生成的密钥，返回服务器
    • 服务器用私钥解密得到对称加密用的密钥
  • 生成数字签名
    • 服务器数据hash生成摘要
    • 服务器私钥加密摘要，生成数字签名
    • 客户端公钥解密得到摘要
    • 客户端对数据hash，判断两个摘要是否相等
  • 生成数字证书
    • 服务器把公钥及自己的信息发给CA，CA用私钥加密，生成数字证书
    • 客户端用CA公钥解密数字证书，得到服务器真实的公钥

A: 1.0.0.0 - 127.255.255.255 （8位网络号） 10.0.0.0 - 10.255.255.255 开头0，网络号是2^7-1个
B: 128.0.0.0 - 191.255.255.255 （16位网络号）172.16.0.0.0 - 172.31.0.0.0 开头10，网络号是2^14-1个
C: 192.0.0.0 - 223.255.255.255 （24位网络号）192.168.0.0 - 192.168.255.255 开头110，网络号是2^21-1个
D: 224.0.0.0 - 239.255.255.255 （多播）开头1110
E: 240.0.0.0 - 255.255.255.255 （保留）开头1111

0.0.0.0表示当前主机，255.255.255.255为当前子网广播地址

java日志

slf4j：
SLF4J用作各种日志框架的壳，允许最终用户在部署时插入所需的日志框架，例如log4j。
private static Logger logger = LoggerFactory.getLogger(Slf4jTest.class);
使用{}而不是字符串拼接

apache-common-logging：
通过动态查找的机制，在程序运行时自动找出真正使用的日志库
private static Log log = LogFactory.getLog(CommonLogTest.class);

log4j：
配置.properties或.xml的文件，配置日志的目的地，格式等等
OFF、FATAL、ERROR、WARN、INFO、DEBUG、TRACE、 ALL
private static Logger log = Logger.getLogger(Log4jTest.class);

logback：
log4j的升级版
Spring Boot 2默认采用了slf4j+logback的形式
